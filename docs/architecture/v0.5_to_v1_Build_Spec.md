# v0.5 → v1 Build Spec

This document is the tracked build specification for the production-grade v0.5 → v1 baseline of the OSINT Investigation Workbench and CASE Binder Builder system.

It is grounded in the existing scaffold direction, the v0.4.3 analyst/control workbench direction, and the unified roadmap for the combined Workbench + CASE Binder platform. fileciteturn28file0 fileciteturn28file2 fileciteturn30file0

---

## 1. Core Principle

```text
Postgres = source of truth
MinIO = raw + derived artifacts
Neo4j = derived graph projection
Celery = execution layer
FastAPI = control plane
React = analyst interface
Binder = first-class subsystem
```

This keeps the canonical data model separate from graph projection and derived outputs.

## 5. Exact System Flow

### Investigation flow
```text
Target created
  ↓
Job requested
  ↓
Celery worker runs connector
  ↓
Raw JSON stored in MinIO
  ↓
Normalization writes canonical records to Postgres
  ↓
Graph projection updates Neo4j
  ↓
Timeline + contradiction + reasoning artifacts generated
  ↓
Frontend shows graph/timeline/claims/artifacts
```

### Binder flow
```text
Evidence roots scanned
  ↓
Manifest created
  ↓
Classification into evidence buckets
  ↓
Exact + near-duplicate detection
  ↓
Master exhibit index
  ↓
Curated packet indexes
  ↓
Page resolver
  ↓
Binder assembly
  ↓
Hearing pack + decision outputs
```

## 6. Functional Baseline by Version

## v0.5 — Stable Platform Baseline
Deliver:
- rooted repo structure
- Docker Compose stack
- FastAPI auth
- Alembic migrations
- Postgres canonical storage
- Redis/Celery execution
- MinIO raw + artifact buckets
- Neo4j graph projection
- working connectors:
  - manual import
  - IP enrichment
  - Shodan/IPInfo class connectors where keys exist
- graph endpoint
- timeline endpoint
- artifacts endpoint
- claims endpoint
- basic frontend shell

### v0.5 Done when
- analyst can create investigation, add target, run jobs, inspect graph/timeline/artifacts
- raw outputs persist
- derived graph/timeline/report artifacts persist
- startup is deterministic

## v1.0 — Production-grade Analyst Baseline
Add:
- RBAC/auth hardening
- saved searches
- analyst dispositions
- richer graph interactions
- stronger contradiction logic
- basic reasoning summaries
- environment split and secrets handling
- audit logging
- binder subsystem skeleton

### v1 Done when
- platform is no longer scaffold-grade
- investigation workflow is coherent end-to-end
- binder system exists as first-class subsystem

## 12. Production Controls

Must-have:
- Alembic migrations only, no `create_all`
- JWT auth now, RBAC by v1
- immutable raw artifact bucket
- artifact provenance fields
- audit log on analyst dispositions and packet builds
- structured logging
- health endpoints
- storage quotas / lifecycle rules
- retry policies on connector runs
- explicit failure state on page-resolution gaps

Nice next:
- SSO
- per-case permissions
- approval workflow for final packet builds
- signed build manifests

---

## 13. Acceptance Criteria

### v0.5 acceptance
- system runs via Docker Compose
- jobs execute asynchronously
- raw results land in MinIO
- normalized records land in Postgres
- graph projects into Neo4j
- frontend shows investigations, graph, timeline, artifacts

### v1 acceptance
- auth and migrations hardened
- analyst review workflow exists
- reasoning summary exists
- binder subsystem exists
- evidence scan/classify/dedupe/index works
- packet build jobs run and produce artifacts
