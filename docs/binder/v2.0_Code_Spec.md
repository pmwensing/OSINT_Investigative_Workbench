# Binder v2.0 Code Spec

This document is the tracked code specification for Binder Subsystem v2.0.

It translates the roadmap and implementation issue into a ready-to-build subsystem spec focused on ingestion, classification, dedupe, and exhibit index generation.

Related tracked items:
- Roadmap: `docs/roadmap/OSINT_Workbench_and_CASE_Binder_Roadmap.md`
- Build spec: `docs/architecture/v0.5_to_v1_Build_Spec.md`
- Issue #6: Binder subsystem v2.0 (ingestion, classify, dedupe, index)

---

## 1. Purpose

Binder v2.0 is the first production-oriented binder subsystem baseline.

Its job is to:
- scan evidence roots
- build a source manifest
- classify evidence into stable buckets
- detect exact and near-duplicate files
- generate a canonical master exhibit index
- generate curated packet indexes for disclosure and adjudicator packet workflows

Binder v2.0 does **not** fully solve page-accurate compiled binder mapping yet. That belongs to the page resolver stage that follows.

---

## 2. Scope

### Included in v2.0
- directory scanning
- source manifest generation
- stable classification buckets
- exact hash dedupe
- near-duplicate grouping heuristics
- canonical exhibit IDs
- master exhibit index CSV
- curated packet indexes CSV

### Not included in v2.0
- final compiled PDF page mapping
- clickable TOC generation
- final binder assembly polish
- full hearing pack generation
- final adjudicator packet formatting

## 4. Data Contracts

### Source manifest row
Fields:
- `source_id`
- `source_path`
- `basename`
- `extension`
- `mime_type`
- `size_bytes`
- `sha256`
- `page_count`
- `discovered_at`
- `scan_root`
- `ingest_status`

### Exhibit index row
Fields:
- `exhibit_id`
- `title`
- `source_id`
- `source_path`
- `category`
- `subcategory`
- `packet_flags`
- `sha256`
- `duplicate_group_id`
- `canonical_preferred`
- `notes`

### Duplicate group row
Fields:
- `duplicate_group_id`
- `match_type` (`exact` or `near`)
- `member_source_ids`
- `preferred_source_id`
- `reason`

## 5. Classification Buckets

Use stable categories:

```text
A_CHRONOLOGY
B_MAINTENANCE
C_PEST
D_FIRE
E_LOCKOUT
F_FINANCIAL
Z_UNCLASSIFIED
```

### Initial classification signals
- filename keywords
- parent folder names
- PDF extracted text when available
- known source patterns
- analyst override hooks later

## 6. Module Responsibilities

### `binder/ingestion/scan_sources.py`
Responsibilities:
- recurse evidence roots
- collect candidate files
- capture basic file metadata
- dispatch PDF page-count/text extraction when available
- emit source manifest rows

### `binder/ingestion/manifest.py`
Responsibilities:
- write `SOURCE_MANIFEST.csv`
- normalize paths
- assign stable `source_id`
- preserve provenance fields

### `binder/classify/rules.py`
Responsibilities:
- define keyword/path/category mapping rules
- expose reusable match helpers
- keep classification rules data-driven where possible

### `binder/classify/classifier.py`
Responsibilities:
- classify sources into category + subcategory
- infer provisional exhibit title
- set packet flags (`full`, `disclosure`, `adjudicator`)

### `binder/dedupe/hash_index.py`
Responsibilities:
- compute exact duplicate groups using SHA-256
- select preferred canonical copy
- emit exact duplicate mapping rows

### `binder/dedupe/near_duplicate.py`
Responsibilities:
- detect probable duplicates where hashes differ
- use heuristics such as filename similarity, size proximity, page count, text similarity
- assign near-duplicate group IDs
- choose canonical preferred member

### `binder/index/build_master_index.py`
Responsibilities:
- combine manifest + classification + dedupe outputs
- assign stable exhibit IDs
- build `MASTER_EXHIBIT_INDEX.csv`

### `binder/index/build_curated_indexes.py`
Responsibilities:
- filter master index by packet flags
- build `DISCLOSURE_EXHIBIT_INDEX.csv`
- build `ADJUDICATOR_EXHIBIT_INDEX.csv`
- support future additional packet variants

## 8. CLI Contract

Provide a starter CLI entrypoint shape like:

```bash
python -m binder.ingestion.scan_sources \
  --root /path/to/evidence \
  --out /path/to/output
```

And a pipeline runner shape like:

```bash
python -m binder.index.build_master_index \
  --manifest /path/to/SOURCE_MANIFEST.csv \
  --out /path/to/output
```

A later unified wrapper can orchestrate the entire v2.0 pipeline.

---

## 9. Acceptance Criteria

Binder v2.0 is complete when:
- evidence roots are scanned reproducibly
- a stable source manifest is created
- files are classified into the defined buckets
- exact duplicates are grouped
- near-duplicates are heuristically grouped
- a canonical master exhibit index is generated
- disclosure and adjudicator packet indexes are generated

---

## 10. Design Notes

- preserve provenance first
- prefer false negatives over destructive false positive dedupe
- do not pretend page accuracy at this stage
- make outputs inspectable and editable by analysts
- keep packet curation deterministic
